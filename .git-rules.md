# Git Rules - Centrum Operacyjne Mieszkańca

## Struktura gałęzi

### Gałęzie główne
- `main` - kod produkcyjny, stabilny
- `develop` - integracja feature'ów

### Gałęzie robocze
```
feature/<opis>    # Nowe funkcje
fix/<opis>        # Naprawy błędów
refactor/<opis>   # Refaktoryzacja
docs/<opis>       # Dokumentacja
chore/<opis>      # Maintenance
```

## Nazewnictwo gałęzi

### Format
```
<typ>/<krótki-opis-kebab-case>
```

### Przykłady
```
feature/weather-widget-api
feature/auth-google-oauth
fix/rss-parser-timeout
fix/duplicate-events
refactor/scrapers-base-class
docs/api-documentation
chore/update-dependencies
```

### Zasady
- Tylko małe litery
- Słowa oddzielone `-` (kebab-case)
- Maksymalnie 50 znaków
- Bez polskich znaków

## Commit messages

### Format
```
<typ>(<scope>): <opis>

[opcjonalnie: dłuższy opis]
```

### Typy
| Typ | Opis |
|-----|------|
| feat | Nowa funkcja |
| fix | Naprawa błędu |
| refactor | Refaktoryzacja |
| docs | Dokumentacja |
| style | Formatowanie |
| test | Testy |
| chore | Maintenance |
| perf | Optymalizacja |

### Scope (opcjonalny)
- `api` - backend API
- `ui` - frontend komponenty
- `db` - baza danych
- `ai` - AI processing
- `scraper` - scrapery
- `auth` - autentykacja

### Przykłady
```
feat(api): add GUS statistics endpoint
fix(scraper): handle Cloudflare timeout
refactor(ai): optimize article categorization prompt
docs: update README with new endpoints
chore: upgrade FastAPI to 0.110.0
```

### Zasady
- Język: angielski
- Pierwsza litera mała
- Bez kropki na końcu
- Tryb rozkazujący ("add" nie "added")
- Max 72 znaki w pierwszej linii

## Workflow

### Feature development
```bash
# 1. Utwórz branch z develop
git checkout develop
git pull origin develop
git checkout -b feature/nazwa-funkcji

# 2. Pracuj i commituj
git add .
git commit -m "feat(scope): opis"

# 3. Push i PR
git push -u origin feature/nazwa-funkcji
# Utwórz Pull Request do develop
```

### Hotfix (pilna naprawa na produkcji)
```bash
git checkout main
git checkout -b hotfix/opis-naprawy
# ... napraw i commituj
git push -u origin hotfix/opis-naprawy
# PR do main + merge do develop
```

## Pull Request

### Tytuł
```
[FEATURE] Krótki opis funkcji
[FIX] Krótki opis naprawy
[REFACTOR] Krótki opis
```

### Opis (template)
```markdown
## Opis
Co zostało zrobione?

## Zmiany
- [ ] Backend
- [ ] Frontend
- [ ] Database migration
- [ ] Documentation

## Testy
Jak przetestować?

## Screenshots (jeśli UI)
```

## Merge Strategy

### develop → feature
- Rebase preferowany
- Squash dla małych feature'ów

### develop → main
- Merge commit
- Tagowanie wersji: `v1.0.0`

## Tagi i wersje

### Format
```
v<major>.<minor>.<patch>
```

### Przykłady
- `v1.0.0` - pierwsza stabilna wersja
- `v1.1.0` - nowa funkcja
- `v1.1.1` - bugfix

## Zabronione

- Force push do `main` lub `develop`
- Commit bezpośrednio do `main`
- Merge bez code review (gdy >1 developer)
- Commity z `.env` lub credentials
- Commity z `node_modules/` lub `.venv/`

## Dobre praktyki

- Małe, atomowe commity
- Opisowe nazwy branchy
- Aktualizacja dokumentacji przy zmianach API
- Testowanie przed merge
- Regularne pull z develop (unikaj konfliktów)
